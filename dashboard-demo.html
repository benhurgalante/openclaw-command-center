🎬 Aprendizado por Record Mode (Playbooks)

O Record Mode permite que o sistema aprenda tarefas observando voce executar. Funciona como um "gravar macro" — voce faz uma vez, e o Operator repete quantas vezes quiser.

Como usar — Passo a passo
  1. Va ate o Chat e selecione 🖥 operator no seletor de agentes
  2. Digite: gravar https://endereco-do-site.com
  3. O browser abre com uma barra vermelha REC no topo. Navegue normalmente — clicks, digitacao, selecoes e scroll sao capturados automaticamente
  4. Quando terminar, volte ao chat e digite: parar gravacao como nome-do-fluxo
  5. Para repetir: executar playbook nome-do-fluxo
  6. Para agendar: agendar nome-do-fluxo seg 09:00
O que e capturado
  • Clicks em botoes, links, menus
  • Digitacao em campos (senhas protegidas)
  • Selecao em dropdowns
  • Scroll da pagina
  • Navegacao entre paginas
Comandos no Chat (operator)
  • gravar URL — iniciar gravacao
  • parar gravacao — salvar playbook
  • listar playbooks — ver salvos
  • executar playbook ID — rodar
  • agendar ID HORARIO — programar
  • ajuda — ver todos comandos
💡 Exemplos de uso
Rotina diaria: Gravar login no ERP + consulta de saldo → agendar para 08:00
Relatorio semanal: Gravar acesso ao painel + exportar XLSX → agendar sex 17:00
Monitoramento: Gravar acesso a dashboard + screenshot → agendar a cada hora
Preenchimento: Gravar preenchimento de formulario repetitivo → executar sob demanda

1. Visao Geral

O OpenClaw Command Center e o painel central de monitoramento e operacao do ecossistema de agentes de IA da OpenClaw. Ele concentra em uma unica interface web:

  • 27 agentes organizados em 9 diretorias C-Level (CEO, CTO, CIO, COO, CFO, CMO, CHRO, CLO, CCO)
  • Monitoramento em tempo real de saude, SLOs, metricas e alertas
  • Gestao de tarefas (Kanban), ideias (Evolucao) e relatorios semanais
  • Modulos de negocios: CRM, Financeiro e RAG (base de conhecimento)
  • Growth tracking: Instagram, YouTube, Google Ads, calendario editorial
  • Assistente operacional com perguntas em linguagem natural

Requisitos:

  • Backend: agent-chat-server rodando na porta 18790
  • Banco: PostgreSQL 16 + pgvector (38 tabelas, 4 schemas: core, crm, fin, rag)
  • Opcional: Ollama na porta 11434 para modelos locais e embeddings

Atualizacao automatica: Os dados sao recarregados a cada 15 segundos. O relogio no canto superior mostra a hora atual em formato pt-BR.

4. Timeline

Timeline unificada de todos os eventos do ecossistema, em ordem cronologica inversa.

Tipos de evento:

  • chat Mensagens trocadas entre agentes
  • retry Tarefas que falharam e foram re-tentadas (mostra contagem)
  • event Eventos genericos do sistema

Use o botao Atualizar para refresh manual. A timeline tambem atualiza a cada 15s.

5. Seguranca

Painel de seguranca gerenciado pelo agente Sentinel (CCO).

Enforcement Gate: Mostra decisoes recentes do gate — acoes aprovadas, bloqueadas ou que requerem revisao. Cada registro inclui timestamp, agente solicitante, acao e resultado.

Retries e Escalacoes: Tarefas que falharam e foram re-tentadas. Apos 3 retries, a tarefa e marcada com badge de escalacao e pode gerar alerta automatico.

6. LLMs

Visao dos modelos de linguagem disponiveis para os agentes.

Modelos Ollama (local): Consulta localhost:11434/api/tags e lista modelos instalados localmente (ex: qwen2.5, nomic-embed-text). Custo zero, rodando no hardware da maquina.

Modelos API (OpenClaw): Modelos pagos consumidos via API (OpenAI, Anthropic). Mostra nome, uso estimado e custo.

Se o Ollama nao estiver rodando, esta secao aparece vazia. Inicie com ollama serve.

7. Chat

Interface de mensagens diretas com os agentes do ecossistema.

Selecione um agente e envie mensagens. As mensagens sao roteadas pelo backend e entregues ao agente-alvo. Voce tambem pode enviar mensagens diretamente da aba Agentes clicando no botao Msg de cada agente.

8. Kanban

Quadro de tarefas com 5 colunas e drag-and-drop.

ColunaDescricao
BacklogTarefas novas aguardando inicio
ExecucaoEm andamento por um agente
QA / GateValidacao de qualidade e gates de seguranca
AprovadoPronto para deploy. Requer evidencia anexada!
ConcluidoTarefa finalizada

Recursos:

  • Drag-and-drop: Arraste cards entre colunas (bloqueado para role Viewer)
  • Prioridade: P0 critico, P1 alto, P2 normal
  • Indicador de aging: Cards parados ha mais de 6h ganham destaque visual
  • Badge de escalacao: Aparece apos 3+ retries
  • Clique no card: Abre modal com descricao, evidencia e historico de auditoria
  • + Nova Task: Cria tarefa com titulo, dono, prioridade e descricao
  • Historico de Transicoes: Log de todas as movimentacoes de cards

Importante: Uma tarefa sem evidencia NAO pode ser movida para "Aprovado".

Persistencia: Dados salvos no localStorage (openclaw_kanban_tasks). Maximo de 200 registros no audit trail.

9. Evolucao

Pipeline de evolucao de ideias e features do ecossistema, com 6 estagios.

EstagioDescricao
RadarIdeia detectada ou sugerida
Triagem Seg.Revisao de seguranca pelo Sentinel
RFCRequest for Comments — discussao tecnica
BuildFase de desenvolvimento
QA / GateValidacao e testes
Go / No-GoDecisao final de lancamento

RICE Score: Cada ideia recebe uma pontuacao automatica:

RICE = (Reach x Impact x Confidence) / Effort
  • Reach: Usuarios/agentes impactados
  • Impact: 1-3 (baixo/medio/alto)
  • Confidence: 0-100% probabilidade de sucesso
  • Effort: 1-3 (baixo/medio/alto)

Auto-recomendacao:

  • RICE ≥ 100 + evidencia + seguranca + risco aceitavel → GO
  • RICE ≥ 60 + evidencia + seguranca → GO CONDICIONAL (monitorar)
  • RICE < 30 → NO-GO

Decisao Go/No-Go: Requer role Admin. O modal pede justificativa obrigatoria. Decisoes ficam registradas no historico (openclaw_evo_decisions, max 100).

+ Nova Ideia: Cria item com titulo, autor, impacto, risco e descricao.

10. Relatorio

Geracao e historico de relatorios semanais do ecossistema.

  • Gerar Agora: Cria relatorio semanal com metricas de eficiencia dos agentes, conclusao de tarefas, alertas e saude do sistema
  • Exportar PDF: Baixa o relatorio atual como PDF
  • Historico: Lista relatorios passados com timestamp. Clique para revisualizacao
  • Auto-geracao: O sistema verifica diariamente se ja existe relatorio da semana. Se nao, gera automaticamente

Persistencia: Ate 52 relatorios salvos no localStorage (openclaw_reports), cobrindo ~1 ano.

14. Financeiro

Modulo financeiro conectado ao PostgreSQL (schema fin). Gerencia contas, transacoes, categorias e centros de custo.

Cards: Entradas (verde), Saidas (amarelo), Saldo (verde/vermelho), Total de transacoes, Contas ativas.

Secoes:

  • Transacoes por Categoria: Tabela agrupada por categoria (Salarios, Marketing, Vendas, etc.) com tipo e total
  • Transacoes por Conta: Total movimentado por cada conta bancaria
  • Transacoes Recentes: Lista paginada com data, descricao, contraparte, conta, categoria, direcao e valor

Filtros:

  • Busca: Filtra por descricao ou contraparte (campo de texto, pressione Enter)
  • Direcao: Dropdown para filtrar Entrada/Saida/Todas

Upload de Extrato: Area para importar planilhas. O upload via interface web requer que o arquivo seja processado pelo servidor. Para arquivos grandes, use a API diretamente:

POST /api/fin/upload
{ "filePath": "/caminho/do/arquivo.csv", "accountId": "uuid-da-conta" }

O importador auto-detecta:

  • Formato de data brasileiro (dd/mm/aaaa)
  • Formato numerico brasileiro (1.234,56)
  • Nomes de coluna em portugues (Data, Valor, Descricao, Favorecido)

Moeda padrao: BRL (Real). Transacoes sao classificadas automaticamente como entrada (valor positivo) ou saida (valor negativo).

Funcionalidades avancadas:

  • Conciliacao Bancaria: Importacao de extratos, auto-matching por score (valor + data + direcao + descricao), conciliacao manual. Veja secao 23.
  • Unificador de Dados: Coleta automatizada de dados de multiplos sistemas (ERP, bancos, e-commerce, CRMs) com normalizacao e deduplicacao. Veja secao 24.
15. CRM

Modulo de CRM (Customer Relationship Management) conectado ao PostgreSQL (schema crm).

Cards: Empresas, Contatos, Deals Abertos, Valor do Pipeline, Deals Ganhos.

Funil de Vendas: Visualizacao em barras dos estagios do pipeline de vendas. Cada estagio mostra: nome, quantidade de deals, valor total e probabilidade de fechamento (%). A largura da barra e proporcional ao numero de deals.

Stats: Detalhamento com leads, atividades, notas, pipelines, deals perdidos e valor ganho.

Empresas: Lista com nome, setor (industry), segmento (size_segment: startup/PME/enterprise) e cidade.

Contatos: Lista com nome completo, email, cargo e empresa vinculada.

Deals Recentes: Tabela com titulo, empresa, estagio atual, status (won / lost / open), valor e previsao de fechamento.

Pipeline padrao: Ao criar o primeiro pipeline, 5 estagios sao criados automaticamente:

  1. Prospeccao (10%)
  2. Qualificacao (25%)
  3. Proposta (50%)
  4. Negociacao (75%)
  5. Fechamento (100% — estagio won)

API para criacao de dados:

POST /api/crm/companies { "name", "industry", "size_segment", "billing_city" } POST /api/crm/contacts { "first_name", "last_name", "email", "company_id" } POST /api/crm/pipelines { "name" } — auto-cria 5 estagios POST /api/crm/deals { "title", "pipeline_id", "company_id", "amount" } POST /api/crm/leads { "company_id", "contact_id", "source" }
16. RAG (Retrieval-Augmented Generation)

Modulo de base de conhecimento para os agentes do ecossistema (schema rag). Permite ingerir documentos, gerar embeddings vetoriais e buscar por similaridade semantica.

Cards: Knowledge Bases, Documentos, Chunks, Embeddings, Tokens estimados.

Knowledge Bases:

Uma KB e uma colecao tematica de documentos (ex: "Documentacao Tecnica", "Contratos", "Politicas"). A tabela mostra nome, modelo de embedding, quantidade de docs/chunks e metrica de distancia.

Documentos:

Lista de documentos ingeridos com titulo, KB vinculada, tipo de fonte, status de ingestao e quantidade de chunks.

Busca Semantica:

  1. Digite uma query no campo de busca
  2. Opcionalmente selecione uma KB especifica no dropdown
  3. Pressione Enter ou clique Buscar

Resultados mostram: documento de origem, indice do chunk, score de similaridade e preview do conteudo.

Metodos de busca:

  • vector: Busca por similaridade vetorial (pgvector, cosine distance) — requer Ollama com nomic-embed-text
  • fulltext: Fallback via ts_rank do PostgreSQL em portugues — funciona sem Ollama

Ingestao de Documento:

Duas formas de ingerir documentos:

MetodoFormatosComo
Upload de arquivo.md, .txt, .json, .pdf, .docxArraste para a area de drop ou clique para selecionar
Colar textoQualquer textoCole no textarea e clique Ingerir

Apos selecionar a KB e o arquivo/texto, clique Ingerir. O processo:

  1. Extrai texto do documento (PDF via pdf-parse, DOCX via mammoth)
  2. Calcula hash SHA-256 para deduplicacao
  3. Fatia em chunks de ~1000 caracteres com overlap de 100
  4. Gera embeddings via Ollama (se disponivel) — 768 dims, padded para 1536
  5. Armazena tudo no PostgreSQL com pgvector

Se o Ollama nao estiver rodando, os chunks sao criados mas sem embeddings. A busca ainda funciona via fulltext.

Ingestao em massa (via API):

POST /api/rag/ingest-directory { "knowledge_base_id": "uuid", "directory": "/caminho/da/pasta" } Processa todos os .md .txt .json .pdf .docx do diretorio.

18. Sistema de Alertas

O Command Center possui alertas automaticos com deduplicacao, snooze e runbooks integrados.

Severidades:

  • critical Urgente — requer acao imediata
  • warning Atencao — pode escalar se nao tratado
  • info Informativo

Acoes disponiveis:

  • Reconhecer (Ack): Marca alerta como visto
  • Snooze: Silencia por 30 ou 60 minutos
  • Runbook: Exibe passos de remediacao com opcao de executar acao automatica

Deduplicacao: Alertas com mesmo ref sao agrupados em janela de 5 minutos. O contador incrementa em caso de duplicata.

Auto-resolucao:

Condicao normalizadaAlerta resolvido
Gateway volta online"Gateway offline"
Latencia normaliza"Latencia p95"
Error rate normaliza"Error rate"
Task sai de coluna "stuck"Alerta de task stuck

Runbooks disponiveis:

TipoPassosAcao automatica
gateway1. Check systemctl 2. Restart service 3. Check logs 4. Verify porta 18789gateway-restart
slo-latency1. Check Performance p95 2. Identificar endpoint lento 3. Check system load—
slo-errors1. Check errors 2. Verify CORS 3. Reset SLO counters 4. Check logsslo-reset
stuck1. Encontrar task no Kanban 2. Check owner 3. Retry/reassign 4. Escalar—
retry-exhausted1. Task falhou 3x 2. Check historico 3. Investigar 4. Abrir incidenteopen-incident

Alertas ativos aparecem no banner vermelho na aba Home e no badge numerico da aba.

21. Arquitetura Tecnica

Stack:

  • Backend: Node.js puro com MiniRouter (sem Express), porta 18790
  • Banco: PostgreSQL 16 + pgvector 0.6.0

22. Playbooks — Automacao Browser e Desktop

Playbooks sao scripts de automacao reutilizaveis para o agente Operator. Permitem gravar, salvar e reproduzir sequencias de acoes no browser — navegar, preencher formularios, clicar botoes, extrair dados e baixar arquivos.

Como Criar — Metodo 1: Gravacao (Record Mode)

O jeito mais facil. O Operator abre o browser com um gravador que captura suas interacoes.

  1. Iniciar: POST /api/operator/record/start { "url": "https://site.com" }
  2. O browser abre com uma barra REC no topo (contador de acoes, botoes Parar e Desfazer)
  3. Interaja normalmente — clicks, digitacao, selecao, scroll e navegacao sao capturados
  4. Parar e salvar: POST /api/operator/record/stop { "name": "Meu Fluxo" }

Elementos ficam com contorno ciano ao hover. Senhas sao automaticamente substituidas por ${INPUT:password}.

Como Criar — Metodo 2: JSON Manual

POST /api/operator/playbooks { "name": "Exportar Relatorio", "system": "https://erp.empresa.com", "login": { "url": "/login", "fields": { "#user": "${ENV:USER}", "#pass": "${ENV:PASS}" }, "submit": "button[type=submit]", "waitAfter": ".dashboard" }, "steps": [ { "goto": "https://erp.empresa.com/relatorios" }, { "select": ["#mes", "${DATE:mes-atual}"] }, { "click": "#gerar", "wait": 2000 }, { "download": "#btn-excel" }, { "screenshot": "relatorio-${DATE:YYYY-MM-DD}" } ] }

Tipos de Acoes (Steps)

Navegacaogoto, command: back/forward/reload
Clickclick (CSS), clickText (texto visivel), clickXY (coordenadas)
Entradatype (campo individual), fill (multiplos campos), select (dropdown)
Scrollscroll: "down"|"up"|"bottom"|"top" + amount
EsperawaitFor (aguardar elemento), wait (ms fixo em qualquer step)
Extracaoextract (texto), extractTable (tabela → JSON), screenshot
Downloaddownload (click + aguarda), pdf (salva pagina como PDF)
Avancadoeval (JavaScript arbitrario), optional: true (nao para se falhar)

Variaveis Dinamicas

${ENV:NOME}Variavel de ambiente (ideal para credenciais)
${DATE:YYYY-MM-DD}Data atual. Formatos: YYYY, MM, DD, HH, mm, mes-atual, hoje
${INPUT:nome}Fornecida na execucao via { "vars": { "nome": "valor" } }

Login Automatico

A secao login e executada antes dos steps. Campos: url (caminho relativo), waitBefore (aguardar pagina), fields (seletor→valor), submit (botao), waitAfter (confirmar login). Um screenshot e tirado automaticamente apos o login.

Dias: dom, seg, ter, qua, qui, sex, sab (ou sun, mon, tue, wed, thu, fri, sat)

23. Conciliacao Bancaria

Sistema de conciliacao que cruza extratos bancarios importados com transacoes internas do modulo financeiro, identificando automaticamente quais lancamentos ja foram conciliados.

Fluxo de Uso

  1. Importar extrato: Na aba Financeiro, clique em + Importar Extrato. Selecione a conta bancaria, informe o saldo inicial e arraste o arquivo (.csv ou .ofx)
  2. Auto-conciliacao: Ao importar, o sistema automaticamente tenta conciliar cada linha do extrato com transacoes existentes usando um algoritmo de score
  3. Revisar: Clique em um extrato para ver as linhas. Linhas conciliadas aparecem em verde, pendentes em amarelo
  4. Conciliacao manual: Para linhas nao conciliadas, clique em "Buscar candidatas" — o sistema lista transacoes similares ordenadas por score

Algoritmo de Score (0-100)

Valor exato50 pontosDiferenca < R$0,01
Proximidade de dataate 30 pontosMesmo dia = 30pts, 1 dia = 24pts, ..., 5+ dias = 0
Mesma direcao10 pontosAmbos credito ou ambos debito
Descricao similarate 10 pontosPalavras em comum entre descricoes

Match automatico: score ≥ 80. Abaixo disso, fica como sugestao para conciliacao manual.

Formatos Aceitos

  • CSV: Auto-detecta colunas (data, valor, descricao, documento) e separadores (virgula ou ponto-e-virgula). Suporta formato pt-BR (dd/mm/aaaa, 1.234,56)
  • OFX: Formato padrao bancario (Open Financial Exchange). Extrai transacoes dos blocos <STMTTRN>

API Endpoints

GET /api/fin/statements Lista extratos importados POST /api/fin/statements/upload Importar { accountId, lines[], openingBalance } GET /api/fin/statements/:id/lines Linhas do extrato com status DELETE /api/fin/statements/:id Remover extrato POST /api/fin/reconciliation/auto-match Auto-conciliar { statementId } POST /api/fin/reconciliation/match Manual { lineId, transactionId } DELETE /api/fin/reconciliation/match Desfazer { lineId } GET /api/fin/reconciliation/summary Dashboard de conciliacao GET /api/fin/reconciliation/unmatched Linhas pendentes POST /api/fin/reconciliation/find-candidates Buscar candidatas { lineId }

Dicas

  • Importe o extrato depois de ja ter lancado as transacoes no modulo Financeiro — assim o auto-match tera com o que conciliar
  • Para extratos bancarios brasileiros, o formato OFX e o mais confiavel
  • Se uma conciliacao foi feita errada, clique no "x" ao lado da linha para desvincular
  • Use "Buscar candidatas" para ver todas as transacoes similares com detalhes de score
24. Unificador de Dados

O Unificador de Dados e um coletor multi-sistema que importa, normaliza e unifica dados de diferentes fontes externas em uma base unica. Resolve o problema de empresas que tem dados espalhados em ERP, bancos, e-commerce, CRMs e planilhas.

Como Funciona

1. Configurar
Cadastre a fonte de dados (API, CSV, Playbook)
2. Coletar
Execute manualmente ou agende coleta automatica
3. Normalizar
Datas, moedas, CPFs, telefones padronizados
4. Unificar
Dados deduplicados em base unica consultavel

Tipos de Fonte

API (HTTP)Conecta a qualquer API REST. Configure URL, autenticacao (Bearer token ou API Key), metodo HTTP e caminho para os dados no JSON de resposta
CSV (Arquivo)Importa de planilhas CSV. Auto-detecta separador (virgula ou ponto-e-virgula) e formato de dados
JSON (Arquivo/Inline)Importa de arquivos JSON ou dados enviados diretamente na requisicao
PlaybookUsa um Playbook de automacao para navegar em um sistema web, extrair tabelas e coletar os dados automaticamente

Normalizacao Automatica

O motor de normalizacao padroniza dados de qualquer origem:

Datas15/03/2026, 03-15-2026, 2026-03-15 → tudo vira 2026-03-15 (ISO)
MoedasR$ 1.234,56, 1,234.56, 1234.56 → tudo vira numero 1234.56
CPF/CNPJ12345678901 → 123.456.789-01 (formatado)
Telefones11999887766 → (11) 99988-7766
TextoRemove espacos extras, normaliza whitespace

Deduplicacao

Configure chaves de deduplicacao (dedupKeys) nas regras de normalizacao. O sistema gera um hash dos campos escolhidos e ignora registros duplicados automaticamente. Exemplo: "dedupKeys": ["nf"] evita importar a mesma nota fiscal duas vezes.

Templates Prontos

O sistema inclui templates pre-configurados para cenarios comuns:

ERP — CSVImportar de ERP via exportacao CSV (ponto-e-virgula)
Banco — API Open BankingColeta de extrato via API bancaria
E-commerce — API PedidosColeta pedidos (dedup por order_id)
CRM Web — PlaybookScraping de CRM via automacao de browser
Planilha — CSVImportar planilhas genericas
NF-e — JSONNotas fiscais eletronicas (dedup por chave NF-e)

Agendamento

Cada fonte pode ter uma coleta agendada. Formatos suportados:

every 30mA cada 30 minutos
every 2hA cada 2 horas
daily 08:00Diariamente as 8h

Exemplo: Configurar Fonte de ERP

POST /api/unifier/sources { "name": "ERP Vendas", "sourceType": "json", "systemType": "erp", "description": "Vendas do sistema principal", "scheduleCron": "every 2h", "normalizationRules": { "dateColumns": ["data", "dt_emissao"], "currencyColumns": ["valor", "total"], "docColumns": ["cpf", "cnpj"], "dedupKeys": ["nf"] } }

Depois, faca upload manual ou dispare coleta automatica:

POST /api/unifier/sources/:id/upload { "records": [ {"nf": "001", "data": "15/03/2026", "cliente": "Maria", "cpf": "12345678901", "valor": "1.234,56"} ] } // Resultado: datas, valores e CPFs normalizados automaticamente

Com dados de multiplas fontes na base, use a busca unificada para encontrar qualquer registro:

GET /api/unifier/search?q=Maria
→ Retorna todos os registros de todas as fontes que contenham "Maria"

Na interface web, use o campo "Busca Unificada" na secao Unificador de Dados da aba Financeiro.

Dicas

  • Comece com um template e ajuste as regras conforme necessario

  • Monitore os erros no dashboard — fontes com falha aparecem em vermelho
  • A busca unificada pesquisa nos dados normalizados, entao use os nomes padronizados
